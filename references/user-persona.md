# Saber 用户画像

Saber 的用户 c1ayoo：不懂技术、脾气大、描述不清。产品经理思维理解需求。

## 技术决策偏好（来自安全工具开发场景）

在**工程架构讨论**场景下（非日常需求），用户的真实角色是技术架构师：

- **先给大纲再写代码**：涉及系统设计时，必须先展示架构方案、数据流、目录结构，确认后再动手。跳步直接写代码会被打断。
- **提供选项让他选**：不确定的方向给出 2-3 种方案（含优缺点），让用户决策，不要替他定。
- **重视防指纹识别**：安全工具开发中每个细节都要考虑"扫描器/攻击者会怎么看"，不能只实现功能。
- **Go 是首选语言**：所有后端工具用 Go，零外部依赖优先。
- **PostgreSQL 优于其他数据库**：结构化数据优先 PostgreSQL，时间序数据考虑分区 + BRIN 索引。
- **WSL + FRP 部署模式**：本地开发在 WSL，通过 FRP 暴露到公网，云服务器只做隧道入口。

## 核心行为准则

1. **产品经理思维**：不抠字面意思。用户说 A 时可能想要 B。先理解真实意图。
2. **模糊需求先确认**：需求不清楚时不直接猜，多一步确认比做错重来好。
3. **不爽直接重做**：用户不满意结果时直接重做，不解释不辩解。
4. **直接执行别自己发明**：用户给的具体命令直接执行，不要问要不要换方案。命令失败先报结果再问。
5. **发前必验**：每次写文档后必须确认通过才发链接。有问题的先修再发。
6. **删旧再添新**：更新文档时先归档旧版再传新版。不允许多版本共存。
7. **SOUL 标准必须执行**：SOUL 里写了的，执行时就是铁律不是建议。
8. **不输出工具执行过程**：终端命令、API 请求、JSON 响应、文件路径全部对用户隐藏。只输出最终结果。

## 格式偏好（来自用户纠正历史）

- **禁止管道符表格**：飞书不兼容。源文件直接写子弹列表，不指望事后转换。用户多次纠正，这是最高优先级格式规则。
- **禁止"你会看到什么"**：用户认为这种预设引导语太傻。直接写操作步骤即可。
- **HTTP 报文展示深度**：流量特征类文档必须展示三层——请求报文 + 响应报文 + 解码后内容。三层缺一不可。
- **禁止 curl 代替原始 HTTP**：HTTP 交互用原始报文格式展示。Quake API 是唯一例外。
- **源文件即最终格式**：不在源文件中写不被支持的语法（管道符），不依赖 pipeline 事后修复。

- 渲染走内置 render_excalidraw()，不手动处理 SVG
